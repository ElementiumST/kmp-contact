Сравнение двух реализаций

Таблица сравнивает ru/iva как более монолитную Android-ориентированную реализацию и kmp-contact как более слоистую и модульную реализацию.

ru/iva kmp-contact
Архитектурный стиль Плюсы: быстрый старт разработки, легко собирать функциональность в одном месте, меньше формальных слоев.
Минусы: высокая связанность, смешение UI, логики и эффектов, сложнее локализовать изменения. 		Архитектурный стиль Плюсы: явные слои ui -> presentation -> domain -> data, лучше читается ответственность каждого модуля, проще расширять.
Минусы: выше порог входа, больше файлов и абстракций даже для простой задачи.
Управление состоянием Плюсы: единая точка хранения данных, удобно быстро пробрасывать состояние в экран.
Минусы: глобальное mutable state, трудно понять кто и где меняет данные, растет риск побочных эффектов. 		Управление состоянием Плюсы: состояние локализовано во ViewModel, используются StateFlow и явные методы событий, поток обновлений предсказуем.
Минусы: нужно поддерживать больше явных моделей состояния и методов обновления.
Связь UI и логики Плюсы: минимальная дистанция между экраном и действием, быстро править поведение.
Минусы: UI сильнее завязан на внутреннюю логику и глобальные контроллеры, труднее переиспользовать и тестировать экран отдельно. 		Связь UI и логики Плюсы: UI в основном отображает state и отправляет callbacks, меньше знания о деталях реализации.
Минусы: для простых сценариев может казаться избыточным количество прослоек между кнопкой и результатом.
Работа с бизнес-логикой Плюсы: логику можно быстро писать рядом с feature-кодом.
Минусы: бизнес-правила со временем расползаются по экранным файлам и helper-функциям, труднее переиспользовать между платформами. 		Работа с бизнес-логикой Плюсы: use case и domain contract делают сценарии явными, бизнес-логика отделена от платформы и UI.
Минусы: часть use case может выглядеть слишком тонкой оберткой над репозиторием.
Доступ к данным Плюсы: быстро организовать прямой вызов сети или локальной логики без большого количества договоренностей.
Минусы: сетевой и экранный код сближаются, сложнее безопасно менять API, кеш и fallback. 		Доступ к данным Плюсы: репозиторий централизует orchestration, маппинг, кеш и fallback, нижний слой владеет источниками данных.
Минусы: чтобы понять полный сценарий, иногда нужно пройти через несколько файлов.
DI и создание зависимостей Плюсы: можно быстро стартовать без сложной конфигурации контейнера.
Минусы: зависимости часто менее явные, больше скрытых связей, тяжелее заменять реализации. 		DI и создание зависимостей Плюсы: Hilt и явные провайдеры делают граф зависимостей прозрачнее, проще подменять реализации.
Минусы: больше инфраструктурного кода и необходимости поддерживать wiring.
Тестируемость Плюсы: отдельные utility-функции тестировать можно.
Минусы: глобальный state, платформенные зависимости и скрытые эффекты усложняют изолированные тесты и mocks. 		Тестируемость Плюсы: контракты Repository, UseCase, Executor лучше подходят для unit-тестов и подмены зависимостей.
Минусы: тестовый набор нужно продумывать по слоям, а не только по экрану.
Жизненный цикл и корутины Плюсы: проще быстро запускать фоновые задачи.
Минусы: при использовании глобальных scope и глобальных объектов сложнее контролировать отмену, утечки и реакцию на lifecycle. 		Жизненный цикл и корутины Плюсы: viewModelScope и scoped-компоненты лучше соответствуют lifecycle, поведение предсказуемее.
Минусы: больше ограничений на то, где и как запускать операции.
Масштабирование кодовой базы Плюсы: на маленьком проекте можно двигаться очень быстро.
Минусы: с ростом экранов и сценариев резко дорожают изменения, поиск причины бага и оценка влияния. 		Масштабирование кодовой базы Плюсы: цена изменений растет медленнее за счет модульности и разделения ответственности.
Минусы: первые этапы разработки могут идти чуть медленнее из-за архитектурной дисциплины.
Порог входа для нового разработчика Плюсы: можно быстро начать править, если понять несколько центральных файлов.
Минусы: реальная логика часто неявно связана и знание проекта накапливается через контекст в голове. 		Порог входа для нового разработчика Плюсы: по структуре проще понять, где искать domain, data, UI и platform-specific код.
Минусы: новичку сначала нужно освоить архитектуру, DI и границы слоев.
Переиспользование между платформами Плюсы: если проект только Android-only, можно не платить за мультиплатформенную архитектуру.
Минусы: повторное использование для web или iOS почти не заложено архитектурно. 		Переиспользование между платформами Плюсы: shared-модули kmp/domain, kmp/data, kmp/support дают базу для переноса сценариев между платформами.
Минусы: не вся логика одинаково удобна для каждой платформы, иногда требуется дополнительная адаптация.
Скорость прототипирования Плюсы: обычно выше на короткой дистанции, потому что меньше формальных ограничений.
Минусы: часть скорости потом возвращается техническим долгом. 		Скорость прототипирования Плюсы: результат чаще сразу ближе к долгоживущему production-коду.
Минусы: для прототипа может ощущаться медленнее из-за слоев и архитектурных решений.
Поддержка в долгую Плюсы: приемлема, пока команда маленькая и проект ограничен.
Минусы: зависимость от автора, который все помнит, выше риск регрессий и дороже рефакторинг. 		Поддержка в долгую Плюсы: лучше подходит для долгой поддержки, роста команды и эволюции требований.
Минусы: требует дисциплины, иначе архитектурные границы начнут размываться.

Короткий вывод: первая реализация выигрывает в локальной скорости разработки, а вторая - в управляемости сложности и поддержке по мере роста системы.
